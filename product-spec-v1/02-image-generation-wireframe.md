# 02 · 图片生成页高保真线框

## 1. 设计方向

借鉴 Higgsfield 的“内容即画布、生成器悬浮、参数渐进展开”用户旅程，但不复制其品牌识别。V1 使用更克制的内部工作室视觉：深色无界画布、冷调蓝紫主色、低噪声层级和快速微交互。

设计关键词：`cinematic`、`quiet`、`dense but breathable`、`instant`。

## 4. 墙面缩放器

### 4.1 Higgsfield 实测

右上控件不是浏览器 zoom，也不是连续的 CSS 卡宽滑杆，而是一个五档图片密度选择器：

- 外层约 172×32px，内部轨道约 144×12px，thumb 为 12×12px。
- 无常驻文字、加减号或数值，只显示细轨道和圆点。
- DOM 使用自定义 Slider 结构，thumb 的 `role="slider"`，`aria-valuemin=0`、`aria-valuemax=4`，共有 0～4 五档。
- 密度值改变时只重排图片墙，不缩放导航、生成器和页面文字。

在 2488px 宽视口、2px 卡片间距下的实测：

| 值 | 首行方图数量 | 首行单卡约宽 | 行为 |
|---:|---:|---:|---|
| 0 | 7 | 351px | 最密，快速扫图 |
| 1 | 6 | 410px | 中高密度 |
| 2 | 6 | 410px | 当前样本与 1 同一分行结果 |
| 3 | 5 | 493px | 大图浏览 |
| 4 | 4 | 617px | 最大细节 |

档位与“列数”不是一一对应关系。滑杆改变的是目标行高；justified 算法按照一组图片的宽高比决定何时截断一行，因此相邻档位可能落在同一个分行阈值，视觉上暂时不变，跨过阈值后整行重新组合。

### 4.2 墙面布局结论

Higgsfield 的墙面不是 Pinterest 式固定列 masonry，而是 Google Photos 式 **justified row gallery**：

1. 图片保持原始宽高比和输入顺序。
2. 从左到右累加图片宽高比，接近当前档位的目标行高时结束该行。
3. 完整行重新计算统一行高，使图片宽度之和 + gap 精确铺满容器。
4. 最后一行不强行拉满，按目标行高左对齐。
5. 改变档位后重新分行，所以同一资产的 x/y、宽高和上下邻居都可能改变。

实测 DOM 结构进一步确认：

- 墙面滚动区是独立的 `overflow-y: scroll` 容器，页面顶栏和生成器不参与滚动。
- 布局容器为 `position: relative` 并写入计算后的总高度。
- 每张卡外层为 `position: absolute`，通过 `transform: translate(x, y)` 和显式 `width/height` 布局。
- 卡片使用 `contain: size layout` 隔离布局影响，图片在卡内 `position: absolute; object-fit: cover`。
- 向下滚动后 DOM 中只保留可见与缓冲卡片，说明墙面使用了虚拟化；未把全部历史资产挂入 DOM。
- 卡片 wrapper 的 CSS `transition-duration` 为 0；实测滑杆值先更新，约 50～250ms 内批量换成新坐标，属于防抖后的快速跳转重排，而不是逐卡 CSS 动画。

### 4.3 V1 实现方案

V1 应复用 justified rows 的视觉逻辑，并改善 Higgsfield 在缩放时可能丢失视觉锚点的问题。

建议五档目标行高（桌面基准）：`180 / 240 / 320 / 420 / 560px`。实际行高由容器宽度和行内图片比例重新计算，不直接把该值作为卡宽。

核心计算：

```ts
rowHeight = (containerWidth - gap * (itemCount - 1))
            / sum(items.map(item => item.aspectRatio))

itemWidth = rowHeight * item.aspectRatio
```

实现步骤：

1. 服务端资产列表必须返回 `width`、`height`；客户端在加载图片字节前即可计算 `aspectRatio`。
2. 主线程以 O(n) 顺序构造 `rows[]`，每行记录 `top`、`height`、`startIndex`、`endIndex`。
3. 对行 top 做二分查找，只挂载视口上下各 1.5～2 屏的行；单次目标为约 40～100 张卡片。
4. 卡片绝对定位：`translate3d(x, y, 0)` + 显式宽高；容器只更新总高度。
5. Slider 使用 0～4 离散值，键盘支持左右方向键；用户偏好只存本地。
6. 拖动时用 `requestAnimationFrame` 合并更新，并以 80ms trailing debounce 提交完整重排。
7. 重排前记录视口中央资产 ID 和其中心偏移；重排后修正 `scrollTop`，保证同一资产仍位于近似位置，而不是只保持滚动百分比。
8. 为美感，只对当前已挂载卡片做 140～180ms FLIP transform 动画；超过 100 张或启用 reduced motion 时立即重排。

禁止使用 CSS Columns：它会造成视觉顺序和 DOM/键盘顺序不一致，也难以实现稳定的虚拟化、原位占位卡和中心资产锚定。

## 5. 图片卡片状态

### 5.0 Higgsfield 实测基线（2026-07-17）

测试条件：Z-Image、1:1、4/4，提交一条中性产品摄影提示词。

```text
T+0s       点击 Generate；生成器保留 prompt 和参数，可继续编辑
T+约5s     墙面第一位一次插入 4 个 1:1 占位卡，原有内容向右/向下重排
生成期间   每卡左上为圆形旋转状态环，右上为圆形取消按钮
           卡面是深灰底 + 顶部柔光渐变，使用缓慢呼吸/流光动效
T+约40s内  4 个占位卡在原尺寸、原位置替换为最终图片
```

页面没有跳转、弹窗或全屏 loading；任务状态直接进入内容墙。DOM 同时暴露了 `Generating` 状态文本，但视觉上以图标为主。实测没有点击取消，因此 Higgsfield 取消后的具体文案和退款/额度语义不作为事实引用。

V1 采用同样的即时内容反馈结构，但使用自己的视觉令牌和任务语义。

### 5.1 成功态

- 图片占满卡片，圆角 10px，无常驻文本边框。
- hover 150ms 后出现自下而上的暗色渐变。
- 右下角显示三个 32px 图标按钮：参考、复制、下载；tooltip 显示中文动作名。
- 左下角可选显示模型短名和比例，仅在 hover 时出现。

### 5.2 排队/生成态

- 创建任务后 100ms 内在墙面最前方为每个预计输出插入与目标比例一致的占位卡；原有图片自然重排。
- 占位卡尺寸从模型比例预计算，后续状态和成图替换不得改变卡片外框。
- 卡面使用 `#1A1D22 → #252932 → #1A1D22` 的低对比纵向渐变；顶部柔光以 2.4s ease-in-out 呼吸，不使用高频闪烁骨架屏。
- 左上放置 30×30px 半透明圆形状态标识：排队为静态时钟弧，提交/生成中为 1.1s 旋转弧。图标旁不常驻文字，但必须有 `aria-label` 和 tooltip：`排队中`、`正在提交`、`生成中`。
- 右上放置 30×30px 圆形取消按钮；图标采用斜杠圆或 ×，tooltip 为 `取消本次抽卡`。
- 只有供应商提供可信进度时才显示百分比；否则使用不确定进度条，不伪造百分比。
- 同一批抽卡以相邻卡组显示，批次间距比卡片间距多 4px，但不增加明显容器。
- prompt 和参数在提交后保留，生成按钮继续可用；用户可以立即开始下一批，不被当前任务锁住。
- 成功时图片在原占位卡内 180ms cross-fade 显示；先完成的图片先显示，不等待整批。

#### 取消交互

- 取消按钮关联 `generation_job`（一次 draw），不是尚未存在的 asset。
- 单击后不弹全屏确认：按钮立刻变为小型 loading，卡面状态改为 `正在取消`。
- 尚未提交上游时真正停止任务；已经提交且供应商不可取消时，停止等待并丢弃迟到结果，同时 toast 提示“上游任务可能仍会消耗额度”。
- 一个 draw 预计产生多张图片时，这些占位卡共享同一 job。hover 任一取消按钮时同组卡片同时描边，tooltip 明确为 `取消本次抽卡（预计 4 张）`，避免误以为只取消单张。
- 取消成功后卡片淡出并在 240ms 内收拢；任务抽屉仍保留 `已取消` 记录。

### 5.3 部分成功/失败态

- 已成功图片立即可用，不等待同批其他任务。
- 失败卡保留在原位，显示简短原因和“重试本次”按钮。
- 鉴权/余额/限流等系统性错误在生成器上方显示一次，不对每张卡重复轰炸。

## 6. 生成器组件规格

### 6.1 提示词

- placeholder：`描述画面、主体、构图、光线与风格…`
- `Cmd/Ctrl + Enter` 提交；Enter 换行。
- 字数上限取当前模型能力；接近上限时显示计数。
- 切换模型时保留提示词和兼容参数，不兼容参数被重置并给一次说明。

### 6.2 参考图

- 入口支持本地上传和从墙面“参考”。
- 缩略图 48×48px，展示上传、完成、失败状态；hover 可移除。
- 超过模型上限时不允许继续添加，并说明当前模型最多支持几张。
- 当模型不支持图生图时，参考图控件禁用；切换到该模型前提示是否清空已有参考图。

### 6.3 模型选择器

```text
┌────────────────────────────────────────┐
│ 🔎 搜索模型…                           │
├────────────────────────────────────────┤
│ 最近使用                               │
│ ◉ Midjourney 8.1      Legnext   4张/次 │
│ ○ FLUX.1 Kontext      OpenRouter 1张/次│
├────────────────────────────────────────┤
│ 全部模型                               │
│ ○ Seedream …          文生图 · 图生图  │
└────────────────────────────────────────┘
```

- 首屏只显示已启用模型；模型数超过 8 时搜索框常驻。
- 每项显示模型名、能力标签、一次抽卡预计输出数，不显示供应商定价。
- 管理员设置的排序优先，个人最近使用次之。

### 6.4 比例、分辨率、抽卡次数

- 比例使用可视化比例块 + 文本；仅展示模型支持项。
- 分辨率仅展示能力协议可选项。若模型不允许显式选择，显示只读 `模型默认`。
- 抽卡次数范围默认 1～4；受模型、供应商并发和全局上限共同约束。
- 参数行实时显示 `预计 N 张`。例如 Midjourney `抽卡 2 × 每次 4 张 = 预计 8 张`。
- 生成按钮文案固定为 `生成`，不加入积分、价格或套餐信息。

## 9. 响应式

- ≥1200px：完整悬浮生成器，参数单行。
- 768～1199px：生成器左右 16px，参数允许换两行，墙面默认卡宽 220px。
- <768px：顶栏简化；生成器贴底全宽；参数进入横向滚动条；墙面 2 列；hover 动作移入预览层。
- V1 优先保证桌面 1280～1920px；移动端做到可用，不追求完整生产力体验。

## 10. 可访问性与性能验收

- 所有图标按钮有可见 tooltip 与 `aria-label`；键盘焦点 2px accent 外环。
- 文字/背景对比满足 WCAG AA；状态不只依赖颜色。
- 图片使用 AVIF/WebP 缩略图、`srcset`、懒加载；原图不进入墙面 DOM。
- 滚动时不触发图片布局抖动：服务端必须返回宽高和占位比例。
- 2000 个资产的墙面只挂载约 80～150 个可见/缓冲节点。
